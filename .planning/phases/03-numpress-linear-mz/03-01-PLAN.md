<objective>
Add Numpress-linear m/z compression to the chunked spectra_data layout and make it the new default.
A vendored, pure-managed C# MSNumpress port encodes each chunk's sorted m/z into a `mz_numpress_linear_bytes`
list<uint8> column (7th chunk field), `mz_chunk_values` becomes null, `chunk_encoding = MS:1002312`, and the
intensity list stays the plain lossless `large_list<float>`. `--no-numpress` / `--lossless` falls back to the
Phase-2 delta chunks (bitwise-L1); `--point` falls back to the v1 point layout (bitwise-L1).

LOCKED scope: intensity stays lossless f32. We do NOT implement intensity Numpress-SLOF (the reference does;
that drives its ~63% total reduction). Our numpress output is therefore a VALID m/z-only-numpress variant of
the chunk schema, intentionally NOT byte/schema-identical to small.numpress.mzpeak. Intensity-SLOF is backlog.

Purpose: smallest m/z encoding (the size win), self-describing transform, bounded L2 m/z fidelity while
intensity remains exact. Output: the MSNumpress codec + numpress chunk emission + CLI default + L2/3-mode
verification.
</objective>

    - OptimalLinearFixedPoint(double[]) is MAX-VALUE based (NOT least-squares): seed maxDouble = max(data[0],data[1]);
      for i>=2, extrapol = data[i-1] + (data[i-1]-data[i-2]), diff = data[i]-extrapol,
      maxDouble = max(maxDouble, ceil(abs(diff)+1)); return floor(0x7FFFFFFF / maxDouble).
      dataSize==0 -> 0; dataSize==1 -> floor(0x7FFFFFFF / data[0]).
    - EncodeLinear(double[], double fp): bytes 0-7 = fp as a BIG-ENDIAN IEEE-754 double (encodeFixedPoint);
      then data[0] and data[1] each as a 4-byte LITTLE-ENDIAN int32 = (long)(value*fp + 0.5) (bytes 8-11, 12-15,
      LSB first); then for i>=2: predicted = ints[1] + (ints[1]-ints[0]) on the QUANTIZED ints,
      diff = (int)(quant[i] - predicted), encoded via encodeInt's 4-bit half-byte scheme starting at byte 16,
      two half-bytes packed per byte, trailing odd half-byte flushed as (hb<<4). EncodeLinear(double[]) overload
      computes fp via OptimalLinearFixedPoint internally. (NOTE: values are 4-byte int32, NOT 5-byte.)
    - encodeInt/decodeInt: the canonical half-byte (nibble) scheme — head nibble encodes leading-zero count
