<objective>
Refactor the mzPeak writer so it converts arbitrarily large RAW files in constant memory and survives individual bad scans — WITHOUT changing the v1 point layout, so the work is verifiable against the v1 conformance bar (mzpeak-validate 0/0, identical logical output).

Two mechanical pillars, both fully specified by RESEARCH.md:
1. Streaming: extend MzPeakParquet with a multi-row-group Open→WriteRowGroup→Close path (verified: one ParquetWriter + N CreateRowGroup() reads back as one logical table). Stream the unbounded DATA facets (spectra_data, spectra_peaks, chromatograms_data) to seekable temp files in bounded row groups (cap 1,048,576 rows), then bounded CopyTo each temp file into its STORED zip entry and delete it. Metadata facets stay buffered (bounded by spectrum count). The seekable temp file is mandatory: a ParquetWriter on a non-seekable stream SILENTLY buffers the whole facet in RAM (no throw) — the trap this plan must structurally prevent.
2. Robustness: make each scan all-or-nothing. The current ROB defect is NOT a missing try/catch (one exists at MzPeakSpectrumWriter.cs:142); it is that points are appended (lines 154-176) BEFORE the metadata record + ordinal++ (223-225), so a mid-scan throw orphans points under an ordinal with no metadata row and reuses that ordinal. Fix: stage each scan's data/peaks/metadata rows locally and commit (append + ordinal++) only on full success; on failure log + ParseInput.NewError() and skip (ordinal NOT consumed), mirroring MzMlSpectrumWriter.

Purpose: operational foundation (constant memory + fault tolerance) that the chunked/Numpress phases build on.
Output: extended MzPeakParquet streaming API, refactored MzPeakSpectrumWriter, new NUnit tests, and recorded MEM-02 peak-RSS measurement.
</objective>
